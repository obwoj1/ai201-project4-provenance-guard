# Provenance Guard

**AI201 Project 4 | Student: obaw**

A Flask backend that any creative-sharing platform could plug into to classify
whether submitted text was written by a human or generated by AI. It combines a
**Groq LLM signal** with **pure-Python stylometric heuristics**, returns an
honest **confidence score** and a plain-language **transparency label**, records
every decision to a structured **audit log**, and gives creators an **appeals**
path when they believe they were misclassified.

📄 Design spec & architecture diagram: [planning.md](planning.md)

---

## Architecture Overview — the path a submission takes

```
POST /submit → groq_signal ┐
                           ├→ combine_scores → get_label → audit log → response
              stylo_signal ┘
```

A creator submits `{ text, creator_id }` to `POST /submit`. The text is fanned
out to **two independent signals**: the **Groq LLM** (semantic — does it *read*
as AI?) and **stylometric heuristics** (structural — sentence-length variance,
vocabulary diversity, punctuation density). Each returns a 0.0–1.0 AI-likeness
score. `combine_scores` blends them (60% LLM / 40% stylometric) into a single
**confidence**; `get_label` maps that to one of three transparency labels. The
result gets a UUID `content_id`, is written to the **audit log**, and returned.
A creator can later contest it via `POST /appeal`, which marks the original
decision `under_review` and stores their reasoning. (Full diagram in
[planning.md](planning.md).)

## Detection Signals

Two **genuinely distinct** signals — one semantic, one structural — so they fail
in different ways and the blend is more reliable than either alone.

### Signal A — Groq LLM (`llama-3.3-70b-versatile`)
- **Measures:** holistically whether the text reads as human or AI (tone,
  phrasing, the over-polished/hedging register typical of AI).
- **Output:** float 0.0 (human) → 1.0 (AI) from a forced-JSON response; falls
  back to **0.5** on any parse/network error so it never crashes or biases.
- **What it misses:** can be fooled by lightly-edited AI, may over-flag
  formal/non-native-human writing, is non-deterministic and opaque.

### Signal B — Stylometric heuristics (pure Python)
- **Measures:** three structural statistics — **sentence-length uniformity**
  (AI is uniform), **type-token ratio** (AI reuses vocabulary), **punctuation
  density** (AI is sparser) — averaged 40/40/20 into one 0–1 score.
- **What it misses:** purely surface-level. Repetitive/simple human writing
  (e.g. poetry) can look AI-like; it understands no meaning.

**Why this pairing:** the LLM understands meaning but is opaque and gameable; the
stylometric signal is transparent and deterministic but shallow. Their errors are
uncorrelated, so blending dampens each one's blind spot.

## Confidence Scoring

`confidence = 0.6 * groq_score + 0.4 * stylo_score` (rounded to 2 dp). The LLM
carries more weight as the stronger judge; the stylometric signal is a
deterministic cross-check that pulls confident-but-wrong LLM calls toward the
middle. We never force a binary at 0.5 — the score is a position on a spectrum,
and a wide middle band lets the system honestly say "we're not sure."

**Thresholds (creator-favoring — see [Known Limitations](#known-limitations)):**

| Confidence | Attribution | Transparency label |
|---|---|---|
| **≥ 0.70** | `likely_ai` | ⚠️ Likely AI-Generated |
| **0.40 – 0.69** | `uncertain` | 🔍 Uncertain Origin |
| **< 0.40** | `likely_human` | ✅ Likely Human-Written |

### How I validated it's meaningful

I ran four deliberately chosen inputs spanning the range and confirmed the
combined score spreads across all three bands (real output from this system):

| Input | groq_score | stylo_score | **confidence** | label |
|---|---|---|---|---|
| Clearly AI-generated | 0.91 | 0.389 | **0.70** | ⚠️ Likely AI-Generated |
| Clearly human-written | 0.21 | 0.315 | **0.25** | ✅ Likely Human-Written |
| Borderline: formal human | 0.41 | 0.509 | **0.45** | 🔍 Uncertain Origin |
| Borderline: lightly-edited AI | 0.42 | 0.352 | **0.39** | ✅ Likely Human-Written |

### Two example submissions with noticeably different confidence

**High-confidence example (0.70):**
> "Artificial intelligence represents a transformative paradigm shift in modern
> society. It is important to note that while the benefits of AI are numerous, it
> is equally essential to consider the ethical implications. Furthermore,
> stakeholders across various sectors must collaborate to ensure responsible
> deployment."
> → `groq_score 0.91`, `stylo_score 0.389`, **confidence 0.70 → ⚠️ Likely AI-Generated**

**Lower-confidence example (0.25):**
> "ok so i finally tried that new ramen place downtown and honestly?
> underwhelming. the broth was fine but they put WAY too much sodium in it and i
> was thirsty for like three hours after..."
> → `groq_score 0.21`, `stylo_score 0.315`, **confidence 0.25 → ✅ Likely Human-Written**

The 0.45-point spread between these confirms the scorer produces real variation,
not a near-constant value.

## Transparency Label — the three variants

The exact text shown to a reader on the platform, by confidence band:

| Result | Verbatim label text |
|---|---|
| **High-confidence AI** (≥ 0.70) | `⚠️ Likely AI-Generated` |
| **Uncertain** (0.40 – 0.69) | `🔍 Uncertain Origin` |
| **High-confidence human** (< 0.40) | `✅ Likely Human-Written` |

Each is a glanceable icon + short phrase. The middle "Uncertain Origin" variant
deliberately makes **no accusation** — it communicates low confidence to a
non-technical reader without exposing a raw number they'd misread.

## Rate Limiting

Applied to `POST /submit` only, via Flask-Limiter (in-memory storage):

```python
@limiter.limit("10 per minute; 100 per day")
```

**Reasoning for these specific values:**
- **10 / minute** — a real creator submits their own work occasionally, not ten
  times a second. Ten per minute is generous for legitimate use (revising and
  re-checking a piece) while a script trying to flood the classifier hits the
  wall almost immediately. The Groq call also costs real quota, so this caps
  runaway usage.
- **100 / day** — an upper bound on sustained automated abuse from one client
  that no genuine individual creator would approach in a day, without blocking a
  power user editing many pieces.

**Evidence (12 rapid requests, limit 10/min):**
```
request  1: 200      request  7: 200
request  2: 200      request  8: 200
request  3: 200      request  9: 200
request  4: 200      request 10: 200
request  5: 200      request 11: 429
request  6: 200      request 12: 429
```
First 10 succeed; the 11th and 12th return **429 Too Many Requests**.

## Appeals Workflow

`POST /appeal` accepts `{ content_id, creator_reasoning }`. It finds the original
decision (404 if unknown), sets its `status` to `under_review`, stores the
reasoning in `appeal_reasoning`, rewrites the log, and confirms receipt. No
automated re-classification — appeals route to a human, who sees the original
decision **and** the creator's reasoning together in the audit entry (filter
`GET /log` on `status == "under_review"`). The first entry in the log sample
above shows a completed appeal.

## Known Limitations

- **Formal writing by non-native English speakers** is the highest-risk
  misclassification. Careful, formal, evenly-paced prose shares surface features
  with AI, so *both* signals can drift upward together — the LLM reads it as
  "too polished," and the stylometric signal sees low sentence-length variance.
  This is a property of the signals themselves, not a data shortage. It's exactly
  why the `likely_ai` bar is high (0.70), the `uncertain` band is wide, and the
  appeals workflow exists — a false positive here is the most damaging outcome on
  a writing platform.
- **Repetitive simple-vocabulary poetry** can be wrongly nudged toward AI by the
  stylometric signal (low type-token ratio + uniform lines), and **very short
  inputs** are statistically meaningless to the heuristics (they return a neutral
  0.5 by design rather than guessing).

## Spec Reflection

- **How the spec helped:** writing the §2 *Uncertainty Representation* section
  first forced me to decide what 0.6 should *mean to a user* before coding —
  which is why the system has a deliberate wide "Uncertain" band and a high AI
  threshold instead of a binary flip at 0.5. Designing the meaning first made the
  thresholds fall out naturally.
- **How implementation diverged:** the spec's example audit entry used a field
  named `llm_score`; I implemented it as `groq_score` to name the specific signal
  source explicitly (and paired it with `stylo_score`). The structure and intent
  are identical — I just chose clearer, signal-specific field names.

## AI Usage

- **Generating the stylometric signal + scorer (M4).** I directed the AI to
  implement `stylometric_signal` (sentence-length variance, type-token ratio,
  punctuation density, no external libraries) and the `combine_scores`/`get_label`
  functions against my spec's 60/40 weights and three thresholds. It produced
  working functions; I verified the threshold **boundaries** against my spec
  (e.g. that exactly 0.70 maps to `likely_ai`, not `uncertain`) and kept the
  neutral-0.5 fallback for too-short inputs.
- **Building the production layer (M5).** I directed the AI to generate the
  `POST /appeal` endpoint and Flask-Limiter setup. I overrode a detail: I made
  the appeal endpoint reject requests missing `creator_reasoning` with a 400
  (rather than silently storing null) so an appeal always carries the creator's
  argument, and confirmed the rate limiter applies to `/submit` only.

## Setup & Running

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env          # then set GROQ_API_KEY=your_actual_key
python3 app.py                # http://127.0.0.1:5000
```

> **macOS note:** AirPlay Receiver also listens on port 5000. If `localhost:5000`
> returns empty/403 responses, either turn off **System Settings → General →
> AirDrop & Handoff → AirPlay Receiver**, or use `127.0.0.1` instead of
> `localhost` in the commands below.

## API quick reference

```bash
# Classify
curl -s -X POST http://127.0.0.1:5000/submit -H "Content-Type: application/json" \
  -d '{"text": "Your text here.", "creator_id": "user-1"}' | python3 -m json.tool

# View the audit log
curl -s http://127.0.0.1:5000/log | python3 -m json.tool

# Appeal a result (use a content_id from a /submit response)
curl -s -X POST http://127.0.0.1:5000/appeal -H "Content-Type: application/json" \
  -d '{"content_id": "PASTE-ID", "creator_reasoning": "I wrote this myself."}' | python3 -m json.tool
```

| Endpoint | Method | Body | Returns |
|---|---|---|---|
| `/submit` | POST | `{text, creator_id}` | `content_id, attribution, confidence, label, groq_score, stylo_score` |
| `/appeal` | POST | `{content_id, creator_reasoning}` | confirmation (404 if id unknown) |
| `/log` | GET | — | `{entries: [...]}` |
