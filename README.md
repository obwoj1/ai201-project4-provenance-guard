# Provenance Guard

AI201 Project 4 | obaw

A Flask backend a creative platform could use to check whether submitted text was
written by a person or generated by AI. It runs the text through two signals (a
Groq LLM and some stylometric math), returns a confidence score and a plain
label, logs every decision, and lets creators appeal when they think the call was
wrong.

Design spec and architecture diagram: [planning.md](planning.md)

## Architecture overview

A creator posts { text, creator_id } to /submit. The text goes to two signals at
once: the Groq LLM, which judges it semantically (does it read like AI?), and the
stylometric heuristics, which look at structure (sentence-length variance,
vocabulary diversity, punctuation density). Each returns a 0-1 AI-likeness score.
combine_scores blends them 60% LLM / 40% stylometric into one confidence number,
and get_label turns that into one of three labels. The result gets a UUID
content_id, gets written to the audit log, and is returned. Later the creator can
contest it with /appeal, which marks the original decision under review and saves
their reasoning. The full diagram is in [planning.md](planning.md).

```
POST /submit -> groq_signal  ┐
                             ├-> combine_scores -> get_label -> log -> response
                stylo_signal ┘
```

## Detection signals

I used two signals that look at different things, so they don't make the same
mistakes.

Signal A is the Groq LLM (llama-3.3-70b-versatile). It reads the text the way a
person would and decides if it feels human or AI-written. I prompt it for JSON and
read its score (0 = human, 1 = AI). If the call or the parse fails it returns 0.5
so nothing crashes and the verdict doesn't get skewed. What it misses: lightly
edited AI can sneak past, and it tends to over-flag very formal human writing. It
also can't explain itself.

Signal B is stylometric heuristics in plain Python. It measures three things and
averages them (40/40/20): how even the sentence lengths are (AI is more uniform),
the type-token ratio (AI reuses words), and punctuation density versus a ~3% human
baseline. What it misses: it only sees surface structure, so repetitive or
simple human writing like poetry can look AI-like to it, and it doesn't understand
meaning at all.

Why both: the LLM understands meaning but is a black box and can be gamed, while
the stylometric one is transparent and consistent but shallow. They fail in
different ways, so blending them covers for each one.

## Confidence scoring

confidence = 0.6 * groq_score + 0.4 * stylo_score, rounded to two decimals. The
LLM gets the larger weight as the stronger judge, and the stylometric score acts
as a check that pulls overconfident LLM calls back toward the middle. I didn't
want a binary at 0.5, so the score is a point on a range with a wide middle band
that lets the system say it isn't sure.

| Confidence | Attribution | Label |
|---|---|---|
| 0.70 and up | likely_ai | ⚠️ Likely AI-Generated |
| 0.40 to 0.69 | uncertain | 🔍 Uncertain Origin |
| under 0.40 | likely_human | ✅ Likely Human-Written |

The AI bar is set high on purpose (see [Known limitations](#known-limitations)):
falsely accusing a real writer is the worse error.

### How I checked it's meaningful

I ran four inputs picked to span the range. Real output from my system:

| Input | groq_score | stylo_score | confidence | label |
|---|---|---|---|---|
| Clearly AI-generated | 0.91 | 0.389 | 0.70 | ⚠️ Likely AI-Generated |
| Clearly human-written | 0.21 | 0.315 | 0.25 | ✅ Likely Human-Written |
| Borderline: formal human | 0.41 | 0.509 | 0.45 | 🔍 Uncertain Origin |
| Borderline: lightly-edited AI | 0.42 | 0.352 | 0.39 | ✅ Likely Human-Written |

### Two examples with different confidence

High confidence (0.70):
> "Artificial intelligence represents a transformative paradigm shift in modern
> society. It is important to note that while the benefits of AI are numerous, it
> is equally essential to consider the ethical implications. Furthermore,
> stakeholders across various sectors must collaborate to ensure responsible
> deployment."

groq_score 0.91, stylo_score 0.389, confidence 0.70 -> ⚠️ Likely AI-Generated

Lower confidence (0.25):
> "ok so i finally tried that new ramen place downtown and honestly?
> underwhelming. the broth was fine but they put WAY too much sodium in it and i
> was thirsty for like three hours after..."

groq_score 0.21, stylo_score 0.315, confidence 0.25 -> ✅ Likely Human-Written

The gap between 0.70 and 0.25 shows the scorer actually varies instead of sitting
near one value.

## Transparency label

The exact text a reader sees, by band:

| Result | Label text |
|---|---|
| High-confidence AI (0.70+) | ⚠️ Likely AI-Generated |
| Uncertain (0.40 to 0.69) | 🔍 Uncertain Origin |
| High-confidence human (under 0.40) | ✅ Likely Human-Written |

Each is an icon plus a short phrase you can read at a glance. The middle one is
worded to never accuse anyone, and none of them throw a raw number at a reader who
might misread it.

## Rate limiting

On /submit only, with Flask-Limiter (in-memory storage):

```python
@limiter.limit("10 per minute; 100 per day")
```

Why these numbers: a real creator submits their own work now and then, not ten
times a second, so 10/minute is plenty for someone revising and re-checking a
piece while a script trying to flood the classifier hits the limit almost right
away. Each submit also spends Groq quota, so the cap keeps costs sane. The 100/day
limit is an upper bound on sustained abuse from one client that a normal person
won't get near in a day.

Evidence (12 fast requests, limit 10/min):
```
request  1: 200      request  7: 200
request  2: 200      request  8: 200
request  3: 200      request  9: 200
request  4: 200      request 10: 200
request  5: 200      request 11: 429
request  6: 200      request 12: 429
```
First 10 go through, 11 and 12 return 429 Too Many Requests.

## Appeals workflow

POST /appeal takes { content_id, creator_reasoning }. It finds the original
decision (404 if the id is unknown), sets its status to under_review, saves the
reason in appeal_reasoning, rewrites the log, and confirms. There's no automatic
re-scoring; an appeal goes to a human who sees the original decision and the
creator's reason in the same entry (filter GET /log for status == "under_review").
The first entry in the log sample above is one that's been appealed.

## Known limitations

The case I'd most expect to get wrong is formal writing by a non-native English
speaker. Careful, formal, evenly-paced prose shares surface features with AI, so
both signals can drift up at the same time: the LLM reads it as too polished, and
the stylometric signal sees low sentence-length variance. That's a property of the
signals themselves, not a lack of data, and it's why the likely_ai bar is high,
the uncertain band is wide, and there's an appeal path. A false positive here is
the most harmful outcome on a writing platform.

Two smaller cases: repetitive simple poetry can get pushed toward AI by the
stylometric signal (low type-token ratio plus uniform lines), and very short
inputs don't give the heuristics enough to work with, so they return a neutral 0.5
instead of guessing.

## Spec reflection

Writing the uncertainty section first made me decide what a 0.6 should mean to a
user before I wrote any code. That's why the system has a wide uncertain band and
a high AI threshold instead of flipping at 0.5. Designing the meaning first made
the thresholds obvious.

Where I diverged: the spec's sample log entry called the model score llm_score. I
named it groq_score instead so it's clear which model produced it, and paired it
with stylo_score. Same structure and intent, just clearer field names.

## AI usage

First, I had the AI write the stylometric signal and the scorer in M4. I told it
to implement stylometric_signal (sentence-length variance, type-token ratio,
punctuation density, no libraries) and combine_scores/get_label against my 60/40
weights and three thresholds. It gave me working functions, but I checked the
threshold boundaries by hand to make sure 0.70 maps to likely_ai and not
uncertain, and I kept the neutral 0.5 fallback for short inputs.

Second, I had it build the production layer in M5: the /appeal endpoint and the
Flask-Limiter setup. I overrode one thing: I made /appeal return a 400 when
creator_reasoning is missing instead of saving a null, so an appeal always carries
an actual argument, and I confirmed the limiter only applies to /submit.

## Setup and running

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env          # then set GROQ_API_KEY=your_actual_key
python3 app.py                # http://127.0.0.1:5000
```

macOS note: AirPlay Receiver also grabs port 5000. If localhost:5000 returns empty
or 403 responses, either turn off System Settings > General > AirDrop & Handoff >
AirPlay Receiver, or use 127.0.0.1 instead of localhost.

## API quick reference

```bash
# Classify
curl -s -X POST http://127.0.0.1:5000/submit -H "Content-Type: application/json" \
  -d '{"text": "Your text here.", "creator_id": "user-1"}' | python3 -m json.tool

# View the audit log
curl -s http://127.0.0.1:5000/log | python3 -m json.tool

# Appeal a result (use a content_id from a /submit response)
curl -s -X POST http://127.0.0.1:5000/appeal -H "Content-Type: application/json" \
  -d '{"content_id": "PASTE-ID", "creator_reasoning": "I wrote this myself."}'
```

| Endpoint | Method | Body | Returns |
|---|---|---|---|
| /submit | POST | text, creator_id | content_id, attribution, confidence, label, groq_score, stylo_score |
| /appeal | POST | content_id, creator_reasoning | confirmation (404 if id unknown) |
| /log | GET | none | entries array |
