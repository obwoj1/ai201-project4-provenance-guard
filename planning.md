# Provenance Guard — Planning

**AI201 Project 4 | Student: obaw**

## Problem

Given a piece of text, decide whether it was written by a human or generated by
an AI model, and expose that decision through a small Flask API with an audit
trail and an appeals process.

## Approach

Combine two independent detection signals so the system does not rely on a single
heuristic:

1. **Groq LLM signal (60% weight)** — send the text to `llama-3.3-70b-versatile`
   with a prompt that forces a JSON response `{"score", "reasoning"}`. The score
   is the model's own estimate of AI-likeness (0.0 human → 1.0 AI). If the
   response cannot be parsed, fall back to a neutral 0.5 so one bad call never
   crashes a request.
2. **Stylometric signal (40% weight)** — pure-Python heuristics, no external
   libraries:
   - **Sentence length variance**: AI prose tends to be uniform; low variance
     (low coefficient of variation) reads as more AI-like.
   - **Type-token ratio**: AI reuses vocabulary; a lower unique/total ratio
     reads as more AI-like.
   - **Punctuation density**: AI prose is often sparser; very low density nudges
     the score up, benchmarked against a ~3% human reference.

The two signals are combined into a single confidence (rounded to 2 dp) and
mapped to a label:

| Confidence | Attribution | Label |
|---|---|---|
| ≥ 0.70 | `likely_ai` | ⚠️ Likely AI-Generated |
| 0.40–0.69 | `uncertain` | 🔍 Uncertain Origin |
| < 0.40 | `likely_human` | ✅ Likely Human-Written |

## Architecture

| File | Responsibility |
|---|---|
| `app.py` | Flask routes, rate limiting, request validation |
| `signals.py` | `groq_signal`, `stylometric_signal` |
| `scoring.py` | `combine_scores`, `get_label` |
| `audit.py` | `write_log`, `read_log` over `audit_log.json` |

## Endpoints

- `POST /submit` — classify text (rate limited 10/min, 100/day)
- `POST /appeal` — flag a classification for human review
- `GET /log` — return the full audit log

## Audit & accountability

Every classification is appended to `audit_log.json` with a UUID `content_id`,
creator id, ISO-8601 timestamp, both raw scores, the combined confidence, a
status, and an appeals field. Creators can appeal a result, which sets the
entry's status to `under_review` and records their reasoning — giving the system
a basic human-in-the-loop accountability mechanism.

## Design decisions

- **Two signals, weighted** — the LLM is the stronger judge (60%) but the
  stylometric heuristics provide a cheap, deterministic cross-check (40%).
- **Safe defaults** — parsing failures return 0.5 rather than erroring.
- **In-memory rate limiting** — sufficient for a single-process demo; would move
  to Redis for production.
- **JSON audit log** — simple and human-readable for a class project; SQLite or
  a real DB would be the production choice.
