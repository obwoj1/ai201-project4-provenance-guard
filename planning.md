# Provenance Guard — Planning

AI201 Project 4 | obaw

I wrote this before building anything. It covers the five design questions, the
architecture and the two main flows, and my plan for how I'd use AI tools in each
milestone.

## Problem and how a submission flows through the system

Creative platforms have a trust problem: a reader can't tell if a post was
actually written by the person who posted it or just generated by AI. The point
isn't to police people, it's to give readers honest context about what they're
looking at. Provenance Guard is the backend piece a platform like that could call.

Here's the path one piece of text takes. A creator sends their text plus a
creator_id to POST /submit. I run that text through two different detectors: a
Groq LLM call that judges it semantically, and some stylometric math that looks
at structure. Each one gives back a number from 0 to 1 for how AI-like the text
is. I combine those two numbers into one confidence value, then turn that into one
of three labels a reader would actually see. The submission gets a UUID, gets
written to the audit log, and comes back to the creator. If they think the call
was wrong, they hit POST /appeal, which marks their entry "under review" and saves
their reasoning next to the original decision.

## Architecture

```
                    SUBMISSION FLOW

  creator
    |  POST /submit  { text, creator_id }
    v
+-------------------------------------------------------------+
|  app.py  (Flask + Flask-Limiter: 10/min, 100/day)           |
|   validates text + creator_id  (400 if either is missing)   |
+-------+------------------------------------------+----------+
        | raw text                                  | raw text
        v                                           v
+----------------------+                  +----------------------+
| groq_signal          |                  | stylometric_signal   |
| (Groq LLM, semantic) |                  | (structural, pure Py)|
|  -> groq_score 0-1   |                  |  -> stylo_score 0-1  |
+----------+-----------+                  +----------+-----------+
           |                                         |
           +------------------+----------------------+
                              v
                  +------------------------+
                  | combine_scores         |
                  |  0.6*groq + 0.4*stylo  |
                  |   -> confidence        |
                  +-----------+------------+
                              v
                  +------------------------+
                  | get_label              |
                  |  >=.70 AI / .40-.69    |
                  |  uncertain / <.40 human|
                  +-----------+------------+
                              v
                  +------------------------+      +----------------+
                  | write_log              |----> | audit_log.json |
                  +-----------+------------+      +----------------+
                              | JSON response
                              v
                           creator


                       APPEAL FLOW

  creator
    |  POST /appeal  { content_id, creator_reasoning }
    v
+-------------------------------------------------------------+
|  app.py                                                     |
|   read_log -> find entry by content_id (404 if not found)  |
|   status -> "under_review";  save appeal_reasoning         |
|   rewrite audit_log.json                                   |
+----------------------------+--------------------------------+
                             | confirmation JSON
                             v
                          creator
```

Submission flow: the text goes to both detectors at once, their two scores get
merged into one confidence, that maps to a label, everything gets logged, and the
result goes back. Appeal flow is smaller: look up the old decision by its id,
attach the creator's reasoning, flip its status, save it.

## 1. Detection signals

I use two signals that look at completely different things, so they don't fail in
the same way.

Signal A is the Groq LLM (llama-3.3-70b-versatile). It reads the text the way a
person would and judges whether it feels human or AI-written. I prompt it to
return JSON with a score and reasoning, and I read the score (0 = human, 1 = AI).
If the call fails or the JSON won't parse, I default to 0.5 so one bad API call
can't crash the request or push the verdict one way. What it misses: lightly
edited AI can slip past it, and it tends to over-flag very formal human writing,
like academic prose or a non-native speaker. It's also a black box, so it can't
really show its work.

Signal B is stylometric heuristics, plain Python, no libraries. It measures three
things and averages them (40% / 40% / 20%): how uniform the sentence lengths are
(AI is more even), the type-token ratio (AI repeats vocabulary more), and
punctuation density compared to a ~3% human baseline (AI tends to be sparser).
What it misses: it only sees surface structure. A repetitive human poem with
simple words can look AI-like to it, and it has no idea what the text actually
means.

I paired them on purpose. The LLM gets meaning but is opaque and can be gamed. The
stylometric one is see-through and consistent but shallow. They make different
mistakes, so averaging them covers for each one's weak spot.

## 2. How I handle uncertainty

The combined score is 0.6 * groq_score + 0.4 * stylo_score, rounded to two
decimals. I give the LLM the bigger share because it's the better judge, and I let
the stylometric score act as a sanity check that drags overconfident LLM calls
back toward the middle.

I didn't want a yes/no answer at 0.5. To me a score of about 0.6 should mean
"leaning AI but not sure," and it lands in the uncertain band on purpose. The
score is a point on a range, and the wide middle band is there so the system can
admit it doesn't know instead of guessing.

Thresholds:

| Confidence | Attribution | What it means |
|---|---|---|
| 0.70 and up | likely_ai | both signals broadly agree it's AI |
| 0.40 to 0.69 | uncertain | mixed or weak signal |
| under 0.40 | likely_human | the default benefit of the doubt |

On a writing platform, calling a real person's work AI is the worse mistake. So I
set the AI bar high at 0.70 and made the uncertain band wide. Borderline writing
gets called "uncertain," not accused. I checked this with four test inputs
(clearly AI, clearly human, formal human, lightly edited AI) and looked at both
the combined and individual scores to make sure they spread out and landed in the
right bands. Results are in the README.

## 3. Transparency label

These are the exact three strings a reader sees:

| Confidence | Label text |
|---|---|
| 0.70 and up | ⚠️ Likely AI-Generated |
| 0.40 to 0.69 | 🔍 Uncertain Origin |
| under 0.40 | ✅ Likely Human-Written |

An icon plus a short phrase is easy to read at a glance, and the band name says
how sure the system is without throwing a raw decimal at someone who'd misread it.
The middle one is worded so it never accuses anyone.

## 4. Appeals workflow

Who appeals: the creator of a classified piece (in a real build, signed in as the
creator_id on that entry). What they send: the content_id of the decision and a
free-text reason for why they think it's wrong. What happens: I find the entry by
its id (404 if it doesn't exist), set its status to under_review, save their
reason in appeal_reasoning, rewrite the log, and send back a confirmation. There's
no automatic re-scoring; appeals go to a human. That human can see the original
decision (both scores, the confidence, the label, the timestamp) and the
creator's reason together in the same log entry, which they'd pull up by filtering
GET /log for status == "under_review".

## 5. Edge cases I expect to handle badly

1. Repetitive, simple poetry. A human poem built on a refrain ("I rise, I rise, I
   rise again") has low sentence-length variance and a low type-token ratio, which
   is exactly what the stylometric signal reads as AI. The LLM usually saves it,
   but a really repetitive piece can get pushed toward likely_ai.
2. Formal writing by a non-native English speaker. Careful, even-paced, formal
   prose looks a lot like AI to both signals at once, so they can drift up
   together. This is the false positive I worry about most, and it's the reason
   the AI threshold is high and the appeal path exists.
3. Very short text. With only a sentence or two there isn't enough to measure, so
   the stylometric signal just returns ~0.5 instead of pretending to know.

## AI tool plan

How I planned to use AI in each implementation milestone, including what spec I'd
hand it and how I'd check its work.

M3 (submit endpoint + first signal): give it the Signal A section and the diagram.
Ask for the Flask skeleton with a POST /submit route and the groq_signal function
returning a 0-1 float with a 0.5 fallback. Check it by calling groq_signal
directly on obvious AI and obvious human samples before wiring it in, then add the
audit log and GET /log.

M4 (second signal + scoring): give it the Signal B section, the uncertainty
section, and the diagram. Ask for stylometric_signal plus combine_scores and
get_label. Check it by running my four test inputs and confirming the scores
spread across all three bands and that the thresholds match exactly (these are
easy for AI to get subtly wrong at the boundaries).

M5 (production layer): give it the label and appeals sections plus the diagram.
Ask for the label function and the POST /appeal endpoint, plus the Flask-Limiter
config. Check that all three labels are reachable, that an appeal flips status to
under_review and saves the reasoning, and that the limiter returns 429 past the
cap.

## Design decisions, short version

- Two weighted signals (60/40): a strong but opaque LLM plus a transparent
  structural check.
- Thresholds favor the creator: high bar for likely_ai, wide uncertain band.
- Neutral defaults: failed parses and too-short text return 0.5.
- JSON log for now because it's readable and simple; I'd use a real database in
  production.
- In-memory rate limiting is fine for one process; production would use Redis.
